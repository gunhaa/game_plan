# Chat, Please — Game Design Document

> 최종 수정일: 2026-05-05  
> 상태: 기획 중

---

## 1. 개요

### 한 줄 소개
> 이 게임은 **버튜버 문화에 공감하는 유저**가 **얼굴 없는 매니저**로서 **실제 채팅 로그 기반의 심사 메카닉**을 통해 버튜버를 성장시키는 **퍼즐 시뮬레이션** 게임입니다.

### 핵심 구조
- **코어**: Papers, Please와 동일한 채팅 심사 메카닉
- **서브**: 버튜버 육성 스토리 (검은 실루엣 매니저와 버튜버의 비밀 관계)
- **확장**: 팬게임 모드 — 실제 버튜버 에셋과 채팅 로그를 넣어 개인화 가능

### 장르
퍼즐 / 시뮬레이션 / 인디

### 타겟 유저
- 버튜버·인터넷 방송 문화에 익숙한 10~30대
- Papers, Please 류의 판단·심사 게임을 즐긴 경험자
- 버튜버 본인 또는 팬덤 (팬게임 모드 타겟)

### 플랫폼
PC (Steam) 우선, 추후 모바일 검토

### 참고 게임
- [Papers, Please](../comparisons/papers_please/overview.md) — 심사 메카닉, 규칙 복잡화 곡선, 도덕적 딜레마 구조

## 2. 핵심 재미 (Core Fantasy)

> "나는 이 버튜버의 채팅창을 통제하는 사람이다."

- **심사의 쾌감**: 규칙을 적용해 올바르게 처리했을 때의 만족감
- **도덕적 긴장**: 규칙상 옳지만 맥락상 애매한 채팅 앞에서 느끼는 갈등
- **비밀의 압박**: 매니저로서 내 정체가 드러나지 않아야 한다는 긴장
- **육성의 보람**: 뒤에서 지켜보며 버튜버가 성장하는 것을 확인하는 뿌듯함

---

## 3. 핵심 루프 (Core Loop)

```
[방송 시작] → 채팅 심사 (허용/삭제/타임아웃/밴) → 돌발 상황 대응
→ [방송 종료] → 결과 피드백 + 버튜버 소통 → 규칙 업데이트 → [다음 방송]
```

### 단기 루프 (방송 1회분 세션 내)
- 채팅이 실시간처럼 스크롤되며 올라옴
- 플레이어는 각 채팅에 대해 빠르게 판단 — 처리 속도와 정확도 사이의 트레이드오프
- 오판 누적 시 경고 → 페널티

### 중기 루프 (수 세션)
- 세션마다 규칙 업데이트 (Bulletin 형식) — 초반 단순 → 후반 복잡
- 버튜버 성장에 따라 채팅량 증가, 심사 난이도 상승
- 성장 지표 확인 (구독자 수, 시청자 수, 후원 수익)

### 장기 루프 (엔드게임)
- 판단 성향 누적 + 버튜버 신뢰 게이지가 엔딩 분기에 영향
- 스토리 이벤트가 특정 세션에 트리거 (서브 요소)

---

## 4. 메카닉

> 상세 내용은 `mechanics/` 폴더 참고 (추후 작성)

### [코어] 채팅 심사
Papers, Please의 서류 심사를 채팅 심사로 대응시킨 핵심 메카닉.

- 4가지 처분: **허용 / 삭제 / 타임아웃 / 영구 밴**
- 현재 규칙 목록은 사이드패널에서 항상 참조 가능
- 유저 이력 조회 가능 (클릭 시 펼침)
- 세션마다 규칙이 1~2개씩 추가·변경 → 점진적 복잡화

**회색지대 채팅** — 도덕적 딜레마의 핵심:
- 규칙 위반이지만 명백한 농담
- 규칙을 교묘히 우회한 악성 채팅
- 버튜버 본인이 방송 중 허용한 발언과 동일한 내용
- 커뮤니티 내부 밈 (외부인에겐 욕설처럼 보임)
- 후원 메시지에 위반 내용이 혼합된 경우

### [서브] 버튜버 어드바이스
- 방송 전·후 버튜버에게 선택지 기반 메시지 전송
- 선택에 따라 버튜버 멘탈 게이지, 신뢰 게이지, 다음 방송 방향에 영향
- 스토리 분기의 주요 트리거

### [서브] 매니저 정체 보안
- 매니저는 항상 검은 실루엣으로 표현, 이름도 "매니저"로만 표기
- 개인정보 노출 선택지는 경고 표시
- 팬덤이 정체를 추측하는 이벤트 → 대응 방식이 스토리에 영향

---

## 5. 채팅 데이터 파이프라인

채팅 심사 메카닉의 원천 데이터를 구성하는 핵심 시스템.

### 데이터 흐름

```
[1. 수집]
실제 인터넷 방송 채팅 로그 수집
(트위치, 치지직, 아프리카TV 등 공개 API 또는 수동 수집)
        ↓
[2. 전처리]
개인정보 제거 · 닉네임 가명 처리 · 중복 제거
        ↓
[3. 카테고리 분류 (라벨링)]
CLEAN / SPAM / MILD_ABUSE / HARD_ABUSE / LINK / GREY / SUPER_CHAT
+ 2차 태그: #meme #context_dependent #evasion #repeat_offender
        ↓
[4. 세션 데이터셋 생성]
시나리오별 규칙·난이도에 맞게 카테고리 비율 조정 후
N개의 세션 데이터셋으로 분기 출력
        ↓
[5. 세션 시퀀스 재생]
타임스탬프 기반 큐레이션 시퀀스로 게임에서 재생
스토리 이벤트용 채팅은 특정 타임스탬프에 강제 삽입
```

### 세션 데이터셋 구성 원칙

| 난이도 | CLEAN | GREY | ABUSE | 총량 |
|--------|-------|------|-------|------|
| 초반 | 70% | 20% | 10% | 200건 |
| 중반 | 55% | 30% | 15% | 350건 |
| 후반 | 45% | 35% | 20% | 500건 |

- GREY 채팅의 정답은 해당 세션의 현재 규칙에 따라 동적으로 결정
- 동일 채팅을 다른 규칙 세션에서 재사용 → 이전에 허용했던 것이 위반이 되는 상황 연출 가능

### 팬게임 모드에서의 데이터 교체
- 스토리 모드의 내장 데이터셋 대신 **외부에서 제공한 채팅 로그**로 교체 가능
- 제공 형식: CSV 또는 JSON (닉네임, 내용, 타임스탬프 컬럼 필수)
- 업로드 시 자동 전처리 (욕설 필터 적용, 라벨 자동 분류) — 수동 수정 가능
- 팬게임 모드는 제공된 로그를 기반으로 세션 데이터셋을 자동 생성

---

## 6. 게임 모드

### 스토리 모드
- 내장 시나리오와 채팅 데이터셋으로 진행
- 매니저(검은 실루엣)와 버튜버 히나의 비밀 관계 스토리 포함
- 엔딩 4개 분기

### 팬게임 모드 (커스텀 에셋)
버튜버 본인 또는 팬이 자신의 에셋과 채팅 로그를 넣어 게임을 개인화하는 모드.

**교체 가능한 에셋:**

| 에셋 항목 | 지원 형식 | 설명 |
|-----------|-----------|------|
| 버튜버 모델 | PNG (투명), Live2D (.moc3) | 메인 캐릭터 비주얼 |
| BGM | MP3, OGG | 방송 배경음악, 루프 설정 포함 |
| 채팅 UI 테마 | JSON (색상·폰트 정의) | 채팅창 색상, 폰트, 배경 스타일 |
| 채팅 데이터 | CSV, JSON | 실제 커뮤니티 채팅 로그 |
| 버튜버 이름 / 팬덤 명칭 | 텍스트 | 게임 내 전역 치환 |
| 규칙셋 | JSON | 커스텀 심사 규칙 정의 |

**팬게임 모드 흐름:**
```
에셋 업로드 → 채팅 로그 자동 분류 → 규칙셋 설정 → 커스텀 세션 생성 → 플레이
```

**공유 기능 (검토 중):**
- 완성된 팬게임 모드 패키지를 단일 파일로 내보내기
- 타인이 해당 파일을 불러와 동일한 환경에서 플레이 가능

## 7. 세계관 & 스토리 (서브 요소)

> 상세 내용은 `narrative/` 폴더 참고 (추후 작성)

### 배경
- 근미래 일본풍 가상 도시, 버튜버 산업이 메이저 엔터테인먼트로 정착한 세계
- 플레이어는 인디 버튜버 **"호시노 히나"** 의 전속 매니저 (비공개 개인 계약)

### 매니저의 정체 — 검은 실루엣
매니저는 게임 내내 검은 실루엣으로만 표현된다. 히나도 매니저의 얼굴을 모른다.  
이 비밀에는 이유가 있다.

### 핵심 반전
> 미결

### 엔딩 분기

| 엔딩 | 조건 | 내용 |
|------|------|------|
| **통합** | 신뢰 MAX + 반전 수락 | 과거를 받아들이고 실루엣이 사라짐 |
| **분리** | 신뢰 HIGH + 반전 거부 | 비밀을 안고 매니저 역할 유지 |
| **붕괴** | 멘탈 LOW | 히나 재은퇴, 매니저도 사라짐 |
| **폭로** | 외부 이벤트 누적 | 팬덤이 정체를 밝혀냄 |

## 8. 콘텐츠 범위

| 항목 | 수량 | 비고 |
|------|------|------|
| 세션(방송 회차) | 20~30 | 챕터 구성 |
| 스토리 이벤트 | 10~15 | 서브 요소, 특정 세션에 삽입 |
| 내장 채팅 DB | 수천 건 | 실제 로그 가공 + 창작 보완 |
| 세션 데이터셋 (N) | 20~30개 | 세션별 독립 데이터셋 |
| 규칙 업데이트 횟수 | 15~20회 | 세션마다 1~2개 추가 |
| 엔딩 | 4개 | 신뢰/멘탈 게이지 + 선택지 기반 |
| 팬게임 모드 커스텀 슬롯 | 6종 | 모델/BGM/테마/채팅DB/이름/규칙셋 |

## 9. 미결 사항 (Open Questions)

- 실제 채팅 로그 수집 방법 및 저작권/개인정보 처리 방안
- 채팅 자동 분류기 구현 방식 (규칙 기반 vs. 경량 ML 모델)
- Live2D 지원 여부 및 PNG 대체 수준 결정
- 팬게임 모드 패키지 공유 플랫폼 (자체 vs. 외부)
- 채팅 심사와 버튜버 어드바이스의 세션 내 시간 비중 조율
- 한국어 방송 로그 vs. 일본어 방송 로그 vs. 혼용 결정
- 스토리라인 핵심 반전
